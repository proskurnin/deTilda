# deTilda — технический обзор проекта (22 апреля 2026)

## 1) Краткое резюме

### Общее состояние
Проект в рабочем состоянии и имеет сильное покрытие ключевых функций (refs/htaccess/html_prettify/forms/checker). Архитектура логически модульная, pipeline читается и покрыт базовыми smoke/регрессионными тестами.

### Сильные стороны
- Единая точка orchestration (`DetildaPipeline`) и последовательные стадии обработки.
- Явные модули по доменам: ассеты, ссылки, формы, htaccess, отчётность.
- Наличие typed-конфига через `ConfigLoader` + `schemas`.
- Хорошие регрессионные тесты по якорям/replace rules/htaccess.

### Главные проблемные зоны
- Хрупкость парсинга `.htaccess` из-за предположения о количестве групп regex и двойного парсинга одного файла.
- Небезопасные режимы скачивания (SSL bypass) и небезопасные defaults в `send_email.php`.
- Риск некорректных замен в refs/assets из-за глобальных regex-подстановок по сырому тексту.
- Непрозрачная/дублирующаяся конфигурационная прослойка (`configuration.py` vs `config_loader.py`).
- Часть тестов красная на текущем состоянии (6 падений).

## 2) Критичные доработки

### 2.1
- **Приоритет**: Critical
- **Модуль / файл**: `core/htaccess.py`
- **Проблема**: `collect_routes()` делает `alias, target = match.groups()` без защиты, если regex не содержит 2 capturing-группы (или содержит 0). Это уже воспроизводится тестами.
- **Риск**: Падение pipeline на реальных `.htaccess` с нестандартными паттернами; не доходит до последующих стадий.
- **Рекомендация**:
  1) Валидировать паттерны из конфига при загрузке (`schemas` + runtime check).
  2) Перед распаковкой groups — проверять `len(match.groups()) >= 2`.
  3) Логировать некорректный паттерн как конфиг-ошибку и пропускать матч, а не падать.

### 2.2
- **Приоритет**: Critical
- **Модуль / файл**: `resources/send_email.php`
- **Проблема**: Хардкод тестового email (`ivan...@gmail.com`) и BCC-списка production-адресов в шаблоне.
- **Риск**: Утечки формы в чужие ящики, юридические/комплаенс-риски, трудно контролируемое поведение на чужом хостинге.
- **Рекомендация**:
  1) Убрать адреса из шаблона.
  2) Делать конфигурацию адресатов только через env (`SENDMAIL_TO`, `SENDMAIL_BCC`, fail-fast если пусто).
  3) Добавить явный security-блок в README.

### 2.3
- **Приоритет**: Critical
- **Модуль / файл**: `core/assets.py`, `core/fonts_localizer.py`
- **Проблема**: Допускается повторная загрузка по HTTPS с отключенной проверкой сертификата.
- **Риск**: MITM и подмена контента (JS/CSS/fonts), компрометация финальной сборки.
- **Рекомендация**:
  1) По умолчанию отключить SSL-bypass (feature flag false).
  2) Включать только явным config флагом + noisy warning + счётчик в финальном отчёте как security incident.
  3) Добавить allowlist доменов и опциональную проверку checksum.

### 2.4
- **Приоритет**: Critical
- **Модуль / файл**: `core/refs.py`
- **Проблема**: `_apply_rename_map()` выполняет глобальный regex replace по всему тексту без контекста (включая JS/CSS строковые литералы, комментарии и т.п.).
- **Риск**: Ложные замены, поломка кода/данных, трудно диагностируемые регрессии.
- **Рекомендация**:
  1) Для HTML менять только значения атрибутов (href/src/action/data-*).
  2) Для CSS — только `url(...)`.
  3) Для JS/JSON — только при явных матчах по ссылочным шаблонам.

### 2.5
- **Приоритет**: Critical
- **Модуль / файл**: `core/assets.py`
- **Проблема**: Приведение относительных ссылок к lower-case делается regex-ом, не учитывающим контекст/OS/edge-cases; возможно изменение путей, которые не должны меняться.
- **Риск**: Тихая порча ссылок, особенно в mixed-case окружениях/инлайн-скриптах.
- **Рекомендация**:
  1) Ограничить lower-case нормализацию только на значения URL в известных атрибутах/функциях.
  2) Добавить dry-run diff режим.
  3) Добавить тесты на false-positive кейсы.

### 2.6
- **Приоритет**: Critical
- **Модуль / файл**: `core/pipeline.py`, `core/checker.py`
- **Проблема**: `stats.broken_links` накапливается из `refs` и затем снова увеличивается результатом `checker`, что может искажать финальную диагностику.
- **Риск**: Неверная метрика качества сборки, ложные тревоги/неверная приоритизация багов.
- **Рекомендация**: Разделить счетчики на `broken_links_detected_during_rewrite` и `broken_links_after_pipeline_check`, а в отчёте выводить обе метрики явно.

## 3) Среднеприоритетные доработки

### 3.1
- **Приоритет**: High
- **Модуль / файл**: `core/configuration.py`, `core/config_loader.py`
- **Проблема**: Две конкурирующие абстракции для работы с конфигом.
- **Риск**: Дублирование логики, дрейф поведения, сложность поддержки.
- **Рекомендация**: Оставить один источник истины (`ConfigLoader` + `schemas`), `configuration.py` либо удалить, либо превратить в thin adapter.

### 3.2
- **Приоритет**: High
- **Модуль / файл**: `core/htaccess.py`
- **Проблема**: Двойной парсинг (`collect_htaccess_routes` + regex-проход в `collect_routes`) пересекается семантически.
- **Риск**: Неочевидные эффекты/дублирование маршрутов/повышенная сложность.
- **Рекомендация**: Свести к единому parser pipeline: parse -> validate -> resolve -> report.

### 3.3
- **Приоритет**: High
- **Модуль / файл**: `core/forms.py`, `core/inject.py`
- **Проблема**: `form-handler.js` генерируется из встроенной giant-string в Python.
- **Риск**: Низкая поддерживаемость, высокий риск ошибок при правках JS.
- **Рекомендация**: Вынести шаблон в `resources/js/form-handler.js` и копировать как статик-ассет; в Python только параметризация.

### 3.4
- **Приоритет**: High
- **Модуль / файл**: `core/assets.py`
- **Проблема**: Потенциальные коллизии имён при переименовании `til -> ai` + sanitize.
- **Риск**: Перезапись/пропуск файлов, неконсистентный rename_map.
- **Рекомендация**: Добавить collision strategy (`name__1`, hash suffix), журнал коллизий в отдельном отчёте.

### 3.5
- **Приоритет**: High
- **Модуль / файл**: `core/inject.py`
- **Проблема**: Проверка «скрипт уже есть» через `if script_name in text`.
- **Риск**: Ложные срабатывания (комментарий/другая строка), либо пропуск нужной инъекции.
- **Рекомендация**: DOM/regex по `<script src=...>` с нормализацией путей.

### 3.6
- **Приоритет**: High
- **Модуль / файл**: `core/report.py`, `manifest.json`
- **Проблема**: В `manifest.json` `features.reports=false`, но код по умолчанию считает отчёты включёнными при отсутствии флага.
- **Риск**: Неочевидное поведение между средами.
- **Рекомендация**: Явный контракт: либо always-on с override, либо strict уважение manifest без fallback.

### 3.7
- **Приоритет**: Medium
- **Модуль / файл**: `core/archive.py`
- **Проблема**: Нет защиты от zip-slip (проверки, что записи zip не выходят из target directory).
- **Риск**: Потенциальная запись файлов вне рабочего каталога.
- **Рекомендация**: Валидировать каждый entry path до extraction.

### 3.8
- **Приоритет**: Medium
- **Модуль / файл**: `core/page404.py`
- **Проблема**: Агрессивное удаление всех `<script>` в 404 может ломать легитимные интеграции.
- **Риск**: Неконсистентная UX/метрики на 404.
- **Рекомендация**: Делать удаление по allow/deny-list и через configurable rule set.

## 5) Тесты, которых не хватает

### Обязательные сценарии
1. `htaccess` с некорректным regex (0/1/3 группы) — pipeline не падает, а репортит ошибку конфига.
2. Коллизии rename (`tilA.png` и `aiA.png`) — корректная стратегия разрешения.
3. `refs` не меняет JS-строки вне URL-контекста.
4. `assets` lower-case не трогает data-uri и JS object keys.
5. `inject` не пропускает инъекцию при ложном `script_name` в комментарии.
6. `send_email.php` поведение без env-адреса (ожидаемый fail-fast).
7. Archive zip-slip payload — блокируется.
8. Checker: отдельные счётчики "до" и "после" rewrite.
9. Remote download timeout/retry/backoff + deterministic logging.
10. Idempotency всего pipeline (двойной запуск не меняет результат).

### Регрессионные тесты (риск-зоны)
- `core/htaccess.collect_routes` при `DirectoryIndex` + custom regex.
- `core/refs.update_all_refs_in_project` для mixed anchors/routes/forms action.
- `core/assets._apply_case_normalization` на Windows-style путях и nested relative paths.
- `core/script_cleaner.remove_disallowed_scripts` на нестандартной разметке script-tag.

## 6) Технический долг

1. Модуль `configuration.py` фактически legacy-дубль поверх `config_loader.py`.
2. Встроенный JS-шаблон в Python (forms) вместо ресурсного файла.
3. Глобальное состояние в логгере и htaccess-кэше (`_routes_info`, `_missing_routes`).
4. Правила замены строк в refs/assets без чёткого AST/DOM-контекста.
5. Security debt: SSL bypass и hardcoded email recipients.
6. Manifest/build/version политика рассинхронизирована (дата релиза 2025-10-24 при свежих изменениях 2026 в README).
